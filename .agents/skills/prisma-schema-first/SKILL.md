# Prisma Schema First Skill

## Purpose

Use this skill whenever the task involves Prisma schema changes, database model changes, migrations, relations, indexes, unique constraints, or generated Prisma client updates.

The goal is to keep Prisma schema as the single source of truth and avoid manually editing migration files.

## Core Rule

Always modify `schema.prisma` first.

Never manually edit files inside `prisma/migrations`.

Migration files must be generated by Prisma using `prisma migrate dev`.

## When to Use This Skill

Use this skill when the user asks to:

* Add, remove, or rename Prisma models.
* Add, remove, or rename fields.
* Add relations between models.
* Add indexes or unique constraints.
* Add enums.
* Update database tables through Prisma.
* Fix Prisma schema errors.
* Generate or apply migrations.
* Change ticket, order, reservation, payment, user, role, permission, or any other database-related model.

## Forbidden Actions

Do not manually edit:

* `prisma/migrations/**/migration.sql`
* old migration folders
* migration history files
* generated Prisma client files
* database tables directly as the first solution

Do not run:

```bash
prisma migrate reset
```

unless the user explicitly approves it.

Do not run:

```bash
prisma db push
```

if the project is already using Prisma migrations.

Do not delete old migrations unless the user explicitly approves it.

Do not patch migration SQL manually to “make it work”.

If migration fails, inspect and fix `schema.prisma` instead of editing migration SQL.

## Correct Workflow

### Step 1: Inspect current Prisma setup

Before making changes, inspect:

* location of `schema.prisma`
* existing models
* existing relations
* existing field naming style
* existing `@@map` and `@map` conventions
* existing migration setup
* package scripts for Prisma commands

### Step 2: Edit only `schema.prisma`

Make all database structure changes in `schema.prisma`.

Follow existing style:

* Use existing model naming convention.
* Use existing ID type convention.
* Use existing timestamp convention.
* Use existing `@map` / `@@map` convention.
* Do not create duplicate models or duplicate relations.
* Do not add fields that conflict with current schema.

### Step 3: Format schema

Run the project-appropriate command, for example:

```bash
pnpm prisma format
```

or if the project requires workspace filtering:

```bash
pnpm --filter @repo/be prisma format
```

### Step 4: Validate schema

Run:

```bash
pnpm prisma validate
```

or the equivalent project command.

If validation fails:

* fix `schema.prisma`
* run format again
* validate again

Do not edit migrations to bypass validation errors.

### Step 5: Generate Prisma Client

Run:

```bash
pnpm prisma generate
```

or the equivalent project command.

### Step 6: Create migration

Only after schema is valid, create migration using:

```bash
pnpm prisma migrate dev --name <meaningful_migration_name>
```

The migration must be generated by Prisma.

Do not manually write or edit the generated SQL unless the user explicitly asks and understands the risk.

### Step 7: Report result

After finishing, report:

* which Prisma models changed
* which fields were added/removed/updated
* which relations were added
* which indexes or unique constraints were added
* the generated migration name
* whether Prisma Client was regenerated
* whether any manual action is still needed

## Error Handling

If Prisma reports:

* migration drift
* failed migration
* checksum mismatch
* database schema is not in sync
* migration history conflict
* shadow database error

Then stop and report the exact problem.

Do not automatically:

* edit old migrations
* delete migrations
* reset database
* run `migrate reset`
* patch SQL manually

Ask the user before doing destructive actions.

## Special Rule for Existing Databases

If the database already has tables and data:

* avoid destructive changes
* avoid dropping columns or tables unless explicitly approved
* prefer adding nullable fields first when needed
* be careful with required fields on existing tables
* explain if a migration may fail because existing rows do not have values

## Special Rule for Production-like Design

For correctness-sensitive data such as tickets, orders, reservations, payments, seats, inventory, and users:

* use database constraints where possible
* use unique constraints for invariants
* do not rely only on Redis or application logic for correctness
* Redis may be used as cache or temporary lock, but the database must remain the source of truth

## Example Instruction

If the user asks:

“Add selected seats to ticket reservation flow”

Then:

1. Inspect existing `Reservation`, `ReservationItem`, `Ticket`, `TicketType`, and `Concert` models.
2. Modify `schema.prisma`.
3. Add new model or fields in schema.
4. Add indexes and unique constraints in schema.
5. Run format, validate, generate.
6. Run `prisma migrate dev --name add_selected_seats`.
7. Do not manually edit migration SQL.

## Final Principle

Prisma schema first.
Migration generated second.
Manual migration editing last resort only with explicit user approval.
